Algorithm:

This web application was written primarily using HTML5, PHP, MySQL, Bootstrap, and was ran locally using the XAMPP server for development.

This application maintains 3 main SESSION objects; currentCourse (holds the id of the current course), currentUnit (holds the id of the current unit), currentLesson (holds the id of the current lesson). This allows for easy access of the user's request. We also implement a quiz for each unit. A user has just one try after which, the quiz submit button is disabled.

Structure:

Most pages in this application, were broken down into parts for easy manipulation, and code reuse. it contains a core folder, which keeps our DB scripts, some helper scripts, and out init scripts. it also contains an include folder which keeps some widgets used on the right menu of the screen, and the upper and lower portion of the page.

Markup:

  • TAGS used are: course, introduction, syllabus, unit, objective, lesson, review, titles, object, ref, research, summary, assignment, books.

  • A course, has the following children: introduction, syllabus, unit, summary, research, assignment, books.

  • A unit, contains the following tags: objective, lesson, review, quiz.

  • A lesson, contains the following tags: title, object

  • Learning objects could either be image or html-text

Parser:

We use PHP SimpleXML parser for parsing. Each lesson object in the lesson table is parsed from top to bottom, and its contains references to either the textobject table, or the imgobject table. Each image reference tag contains a reference attribute, a type, and size. Each text reference contains a type and a reference attribute.

Database:

Our database contains 6 tables namely:

  • courses: holds all information regarding a course. The fields include: courseID, courseTitle, introduction, syllabus, summary, research, books, assignment. The fields holds formatted html markup, that way each time we output a record to the browser, it gets automatically rendered.

  • units: holds all information regarding a unit. The fields include: unitID, unitTitle, objective, review, lesson, quiz. The fields holds formatted html markup, that way each time we output a record to the browser, it gets automatically rendered. We don't use the markup in the lesson field for this app. it was used for debugging purposes. A unitID and it's courseID uniquely identifies a record.

  • lessons: holds all information regarding a lesson within a unit. The fields include: lessonID, lessonTitle, lessonContent, courseID_Ref (FK from courses), unitID_Ref (FK from units) The fields holds formatted html markup, that way each time we output a record to the browser, it gets automatically rendered. The lessonContent field contains references of learning objects in a different table.

  • quizzes: holds all information regarding a quiz for a specific unit. The fields include: quizID, quizContent, quizAnswer, courseID_Ref (FK from courses), unitID_Ref (FK from units. The quizContent field holds formatted html markup, that way each time we output a record to the browser, it gets automatically rendered. Multiple quiz questions related to a specific unit are stored in the table. NOTE: each record holds a quiz question.

  • textobject & imgobject: are two different tables that holds all information for a specific lesson or lessons. In the textobject table, we store its contents in html mark up (p, div etc). In the imgobject table, we store the urls to different images that are stored in a folder structure.
